# Полный аудит backend (FastAPI) — 2026-04-22

## Объект аудита
- Репозиторий: `music_backend`
- Основной файл: `server.py`
- Зависимости: `requirements.txt`

## Краткий итог

Сервис в целом рабочий, с уже внедрёнными базовыми мерами безопасности (проверка Telegram `initData`, CORS, security headers, HTML-экранирование полей). Однако в текущем виде есть несколько **существенных рисков**, из которых два требуют приоритета:

1. **Утечка административных данных** через публичные API.
2. **SSRF-риск в парсере ссылок** (пусть и под админским доступом).

Также есть архитектурные и логические проблемы (идентификатор лайков, mutable default, отсутствие ограничений частоты запросов).

---

## Методика

Проведён статический аудит исходного кода и конфигурации зависимостей с фокусом на:
- аутентификацию/авторизацию,
- безопасность API,
- целостность данных,
- устойчивость к злоупотреблению,
- эксплуатационные риски.

## Найденные проблемы

## 1) [HIGH] Утечка списка администраторов в публичных ответах

**Где:**
- `GET /api/data` возвращает `adminUsernames` для всех клиентов.
- `GET /api/health` возвращает `admins`.

**Риск:**
- упрощение таргетированных атак на админ-аккаунты (социнжиниринг, credential stuffing, фишинг),
- раскрытие внутренней модели прав доступа.

**Рекомендация:**
- убрать `adminUsernames`/`admins` из публичных ответов,
- если нужно — отдавать только админу (через `require_admin`) либо скрывать полностью.

---

## 2) [HIGH] SSRF в `/api/parse_link` при загрузке произвольного URL

**Где:**
- `get_metadata_from_page(url)` выполняет `httpx` GET на любой URL из запроса.

**Риск:**
- доступ к внутренним ресурсам (`localhost`, metadata endpoints, приватные подсети),
- сканирование внутренней сети, обход сетевой сегментации.

**Комментарий:**
- Endpoint защищён админ-доступом, но SSRF остаётся критичным в случае компрометации админ-сессии.

**Рекомендация:**
- запретить private/link-local/loopback диапазоны IP,
- ввести allowlist доменов (например, только Spotify/Яндекс Музыка/Apple Music и др.),
- отключить/жёстко контролировать redirects,
- ограничить размер ответа и тип контента.

---

## 3) [MEDIUM] Нестабильный ключ пользователя в лайках (используется display_name)

**Где:**
- Лайки пишутся по полю `username: user.display_name`,
- чтение лайков также по `display_name`.

**Риск:**
- если `display_name` меняется (например, изменение username в Telegram или fallback на first_name), связь пользователя с лайками ломается,
- возможны дубликаты/потеря управляемости своими лайками.

**Рекомендация:**
- использовать стабильный идентификатор (`authorId`/`user_id`) и/или `username` в нормализованном виде,
- мигрировать индекс `likes(releaseId, username)` в `likes(releaseId, userId)`.

---

## 4) [MEDIUM] Mutable default в модели Review

**Где:**
- `criteria: dict = {}` в Pydantic-модели.

**Риск:**
- общий изменяемый объект между экземплярами (возможные побочные эффекты и трудноуловимые баги).

**Рекомендация:**
- заменить на `Field(default_factory=dict)`.

---

## 5) [MEDIUM] Отсутствие rate limiting на чувствительных endpoint

**Где:**
- Нет ограничений частоты запросов на `/api/reviews`, `/api/likes`, `/api/parse_link`, `/api/block`.

**Риск:**
- abuse/флуд, повышенная нагрузка на БД и внешние сервисы, DoS-поведение.

**Рекомендация:**
- добавить rate limiting (по IP + user_id),
- отдельно ограничить `/api/parse_link` из-за внешних HTTP-запросов.

---

## 6) [LOW] Слишком широкое подавление исключений

**Где:**
- `except Exception:` в нескольких местах (парсинг metadata, AI parser и др.).

**Риск:**
- потеря диагностируемости, скрытие системных ошибок,
- усложнение расследований инцидентов.

**Рекомендация:**
- логировать исключения структурированно,
- ловить более узкие типы ошибок.

---

## 7) [LOW] Security headers частично устарели

**Где:**
- используется `X-XSS-Protection` (legacy).

**Рекомендация:**
- оставить современные заголовки (`CSP`, `Referrer-Policy`, `Permissions-Policy`),
- `X-XSS-Protection` можно убрать как неэффективный в современных браузерах.

---

## 8) [LOW] Отсутствует явное закрытие соединения MongoDB

**Где:**
- клиент создаётся глобально, но нет shutdown-hook с `client_db.close()`.

**Риск:**
- неаккуратное завершение в некоторых сценариях деплоя/перезапуска.

**Рекомендация:**
- добавить `@app.on_event("shutdown")` и закрытие клиента.

---

## Позитивные аспекты

1. Реализована криптографическая проверка Telegram `initData` через HMAC-SHA256.
2. Fail-open устранён: без токена в production запросы не проходят.
3. Есть защита от XSS через `html.escape` в ключевых полях.
4. Установлены MongoDB индексы (включая уникальные).
5. Блокировка пользователей учтена в write-операциях (`reviews`, `likes`).

## Приоритетный план исправлений

### Спринт 1 (срочно)
1. Убрать утечку `adminUsernames`/`admins` из публичных endpoint.
2. Закрыть SSRF в `parse_link` (allowlist + IP restrictions).
3. Перевести лайки на стабильный user key (`userId`).

### Спринт 2
4. Внедрить rate limiting.
5. Исправить mutable default в `Review.criteria`.
6. Добавить структурированное логирование ошибок.

### Спринт 3
7. Актуализировать security headers (`CSP`, `Referrer-Policy`, `Permissions-Policy`).
8. Добавить shutdown-hook для MongoDB.

---

## Вывод

Backend можно считать функционально готовым, но не fully hardened для продакшна с высокой нагрузкой/угрозами. После закрытия пунктов Sprint 1 уровень риска существенно снизится (особенно в части компрометации админ-функций и сетевой поверхности атаки).
